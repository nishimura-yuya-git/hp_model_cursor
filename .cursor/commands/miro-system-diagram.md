# Miro システム設計図 反映コマンド（汎用）

## 1. 実行前に確認すること

Miroへ反映する前に、以下をユーザーへ確認する。

```text
□ MiroボードURL
□ 反映したい領域（全体 / 業務フロー / DB設計 / 外部連携 / UI遷移など）
□ 既存図を更新するか、新しい図として追加するか
□ 図に含めてよい情報（画面名のみ / DB名も可 / API名も可）
□ 顧客や社外に見せる図か、開発者向けの図か
```

固定のMiroボードURLを前提にしない。ユーザーがURLを提示していない場合は、先に確認する。

---

## 2. 実行前に読むもの

Miro MCPを使う場合は、Miroスキルを読む。

```
/Users/yuya/.cursor/plugins/cache/cursor-public/miro/9627168a40c694c61a8734cbcdd452d21e560bdc/skills/miro-mcp/SKILL.md
```

プロジェクト内に業務フロー図や設計資料がある場合は、先に読む。

```text
docs/architecture/**
PROJECT_MEMORY.md
.cursor/rules/database.mdc
.cursor/rules/invariants.mdc
```

## 3. 図にする対象の整理

図を作る前に、以下をチャットへ短く整理する。

```markdown
## Miro反映前の整理

### 対象領域
- ...

### 図に含める画面・処理
- ...

### 図に含めるデータ・外部連携
- ...

### 既存図への影響
- 更新 / 追記 / 新規作成
```

---

## 4. 図の作成方針

### 社外・顧客向け

- 画面名・業務名を中心に書く。
- DBテーブル名・関数名・内部変数名は原則出さない。
- メリットや運用上の意味が伝わる表現にする。

### 開発者向け

- 画面名、API、DB、外部サービス、バッチ、Webhookなどを明示してよい。
- SSoT、確定データ、手動修正保護など、壊してはいけない境界を示す。
- 既存図を壊さず、新規フレームまたは空き領域へ追加する。

---

## 5. Miro操作の基本手順

1. `context_explore` で既存ボードの内容と配置を確認する。
2. 既存図を更新する場合は、差分をユーザーに説明して承認を得る。
3. 新規追加の場合は、既存図と重ならない位置へ配置する。
4. `diagram_create` や `doc_create` で図・補足説明を追加する。
5. 完了後、確認用URLまたは `moveToWidget` URLを報告する。

---

## 6. 禁止事項

- ユーザー確認なしに既存図を削除しない。
- 固定のボードURLを使わない。
- 社外向け図に内部テーブル名や秘密情報を書かない。
- 業務用語を勝手に英語化しない。
- 図の更新で仕様変更が必要だと分かった場合、コード変更に進まず先に確認する。

---

## 7. 報告フォーマット

```markdown
## Miro反映結果

- 対象ボード:
- 作成/更新した図:
- 主な内容:
- 確認URL:
- 追加で確認が必要な点:
```
