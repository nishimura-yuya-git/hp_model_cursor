# PROJECT_MEMORY 学習反映コマンド

## ゴール

`memory-learning.mdc` で検知・提案された「仕様決定」「再発防止」「実装漏れ」「修正未反映」「認識違い」「禁止事項」を、ユーザーの明示指示がある場合だけ `PROJECT_MEMORY.md` に追記する。

このコマンドは、AIが勝手に `PROJECT_MEMORY.md` を書き換えることを防ぎつつ、重要な学習事項を忘れないように残すための安全な入口である。

## 呼び出し条件

以下のように、ユーザーが明示的にこのコマンドを呼び出した場合のみ実行する。

```text
/project-memory-learn
この追記候補を PROJECT_MEMORY.md に入れて
```

```text
/project-memory-learn
今回の修正漏れを再発防止として残して
```

```text
/project-memory-learn
この仕様決定を長期記憶に反映して
```

通常会話で「覚えておいて」「忘れないで」と言われただけでは、このコマンドを実行した扱いにしない。その場合は追記候補を提示し、ユーザーに `/project-memory-learn` の実行を促す。

## 実行前に必ず読むもの

1. `PROJECT_MEMORY.md`
2. `.cursor/rules/memory-learning.mdc`
3. `.cursor/rules/safety.mdc`
4. `.cursor/rules/understanding-first.mdc`
5. `.cursor/rules/change-contract.mdc`

関連する業務コア・DB・計算系の内容であれば、必要に応じて以下も読む。

- `.cursor/rules/database.mdc`
- `.cursor/rules/invariants.mdc`
- `.cursor/rules/architecture-extension.mdc`

## 入力として受け取るもの

ユーザーのメッセージ、または直前の会話から以下を抽出する。

- 分類
- 事象
- 原因
- 今後の対応
- 関連ファイル・関連画面・関連機能
- 実行した修正や確認内容

不足している場合は、推測で埋めずに「要確認」として扱う。

## 分類

追記時は、以下の分類から選ぶ。

| 分類 | 追記先の目安 |
|---|---|
| `仕様決定` | `## 7. 決定済み仕様メモ` |
| `再発防止` | `## 5. 📜 過去に AI が壊した事例（再発防止メモ）` |
| `実装漏れ` | `## 5. 📜 過去に AI が壊した事例（再発防止メモ）` |
| `修正未反映` | `## 5. 📜 過去に AI が壊した事例（再発防止メモ）` |
| `認識違い` | 内容により `## 3. 📐 ドメインルール` または `## 5. 📜 過去に AI が壊した事例` |
| `禁止事項` | 内容により `## 2. 🚫 Hard Boundaries` または `## 3. 📐 ドメインルール` |

判断に迷う場合は、`## 5. 📜 過去に AI が壊した事例（再発防止メモ）` に簡潔に追記し、恒久ルール化が必要なら別途 `.cursor/rules/` への追記提案を出す。

## 実行ステップ

### STEP 1: 既存内容を確認する

`PROJECT_MEMORY.md` を読み、同じ内容がすでに存在しないか確認する。

確認観点:

- 同じ仕様がすでに決定済み仕様メモにあるか
- 同じ失敗が過去事例にあるか
- Hard Boundariesやドメインルールにすでに十分書かれているか
- 既存内容と矛盾していないか

### STEP 2: 追記内容を確定する

以下の形式で、追記内容を簡潔にまとめる。

#### 仕様決定の場合

```markdown
### X.X [機能名・仕様名]（YYYY-MM-DD 決定）

- [決定事項]
- [今後守ること]
- [未決事項があれば要確認として記載]
```

#### 再発防止・実装漏れ・修正未反映の場合

```markdown
- YYYY-MM-DD: [短い事象名]
  - 事象: [何が起きたか]
  - 原因: [分かる範囲。推測なら要確認]
  - 再発防止: [次回AIが必ず行うこと]
  - 関連: `[file]`, `[画面名]`, `[機能名]`
```

### STEP 3: 追記する

ユーザーがこのコマンドを呼び出している場合に限り、`PROJECT_MEMORY.md` を編集してよい。

編集ルール:

- 既存内容を削除しない
- 既存見出しを勝手に変更しない
- 既存番号体系を壊さない
- 追記は必要最小限にする
- 矛盾がある場合は編集せず、ユーザーに確認する
- 推測を確定事実として書かない
- 個人情報や秘匿情報を不要に増やさない

### STEP 4: 完了報告する

追記後、以下を報告する。

```markdown
## PROJECT_MEMORY.md 学習反映完了

### 追記した分類
- ...

### 追記先
- `PROJECT_MEMORY.md` の `...`

### 追記した要点
- ...

### 追記しなかった内容
- なし / ...
```

## 重複時の対応

同じ内容がすでに `PROJECT_MEMORY.md` にある場合は、追記しない。

代わりに以下を報告する。

```markdown
既に同等の内容が `PROJECT_MEMORY.md` にあるため、追記しませんでした。
該当箇所: `...`
```

## 矛盾時の対応

既存内容と新しい追記候補が矛盾する場合は、編集せずに確認する。

```markdown
既存のPROJECT_MEMORY.mdと追記候補が矛盾しています。

既存: ...
追記候補: ...

どちらを正としますか？
```

## 禁止事項

- このコマンドが呼ばれていない状態で `PROJECT_MEMORY.md` を編集すること
- 既存内容をAI判断で削除・上書きすること
- 大量の会話ログをそのまま貼ること
- テストしていない内容を「検証済み」と書くこと
- 原因が不明なのに断定すること
- ユーザーや先方の責任に寄せて書くこと
