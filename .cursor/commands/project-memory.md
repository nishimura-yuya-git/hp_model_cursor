# PROJECT_MEMORY 生成・更新コマンド

## ゴール

ユーザーが提示した要件定義、設計メモ、会話ログ、画面仕様、DB設計案をもとに、プロジェクト横断テンプレートである `PROJECT_MEMORY.md` をその案件向けに整える。

このコマンドは、原則として **`/project-memory` と要件定義書を一緒に渡せば、`PROJECT_MEMORY.md` を完成に向けて作成・更新する** ためのもの。

ただし、「完成」とは推測で全項目を埋めることではない。
要件定義書だけで確定できる項目は反映し、DB設計・権限設計・運用確認が必要な項目は質問または `[DB設計確定後に記入]` として残す。

特に以下を明文化する。

- プロジェクト概要
- 絶対に壊してはいけない領域
- 計算・判定・表示優先度のSSoT
- DBの事実テーブル / 確定テーブル / 集計テーブル
- RLS・権限境界
- マスタ変更時に過去分が塗り替わるか
- 手動修正・確定済みデータの保護
- 未確定の要件・DB設計待ちの項目
- 過去に起きた失敗と再発防止

## 重要方針

要件定義の段階では、DB設計が確定しないと書けない項目がある。

そのため、このコマンドでは **無理に全項目を埋めない**。

以下の3区分で整理する。

```text
確定済み
  → PROJECT_MEMORY.md に反映してよい

要確認
  → ユーザーに質問する

DB設計確定待ち
  → 仮決めせず、未確定事項として残す
```

## 実行前に必ず読むもの

1. `PROJECT_MEMORY.md`
2. `.cursor/rules/database.mdc`
3. `.cursor/rules/safety.mdc`
4. `.cursor/rules/understanding-first.mdc`
5. `.cursor/rules/change-contract.mdc`

既存 `PROJECT_MEMORY.md` に記入済みの内容がある場合は、参考として読む。
ただし、既存プロジェクト固有の内容を別案件へそのままコピーしない。

## 入力として受け取るもの

ユーザーは以下のいずれかを渡す。

- `/project-memory` と要件定義書ファイル
- 要件定義テキスト
- 会話ログ
- 仕様書ファイル
- Miroや業務フロー図の内容
- DB設計メモ
- 既存コード・既存テーブル
- 「この案件用に PROJECT_MEMORY を作って」という自然文

### 要件定義書を受け取った場合の動作

要件定義書を受け取ったら、AIは以下を行う。

1. 要件定義書から確定情報を抽出する
2. DB設計が必要な項目を切り分ける
3. SSoT・Hard Boundaries・業務コア候補を整理する
4. 不足情報を質問として出す
5. `PROJECT_MEMORY.md` への反映案を提示する
6. ユーザー承認後に `PROJECT_MEMORY.md` を更新する

要件定義書に書かれていないテーブル名・カラム名・RLS・計算式は、推測で確定扱いにしない。

## 必須ヒアリング項目

不足している場合は、実装・更新前に質問する。

### 1. プロジェクト基本情報

- プロジェクト名
- 何を実現するシステムか
- 主な利用者
- 技術スタック
- 本番環境
- 最重要業務領域

### 2. 業務コア

- 絶対に壊してはいけない画面・処理
- 1箇所壊れると連鎖的にズレる業務
- 手動修正や例外処理がある箇所
- 確定後に変更してはいけないデータ

### 3. DB・計算系

- 事実データを保存するテーブル
- 計算済みの確定データを保存するテーブル
- 月次・集計・表示用のテーブル
- 計算SSoTとなる関数・サービス
- 複数画面で同じ数値を表示する箇所
- マスタ変更が過去データに影響するか

### 4. 権限・セキュリティ

- 管理者・一般ユーザー・スタッフ・顧客などのロール
- 本人だけが見られるデータ
- 管理者だけが変更できるデータ
- RLSやAPI認可が必要なテーブル

### 5. テスト・運用

- テスト専用ユーザーやテスト用日付
- 触ってはいけない本番データ
- パスワードや認証情報の扱い
- 過去に壊れたことがある処理

## 出力ステップ

### STEP 1: 理解レポートを出す

以下の形式で、まずチャットに出す。

```markdown
## 🔍 PROJECT_MEMORY 反映前の理解レポート

### 1. 読み取った要件
- ...

### 2. 確定済みとして書ける内容
- ...

### 3. DB設計が確定しないと書けない内容
- ...

### 4. 要確認事項
- ...

### 5. PROJECT_MEMORY.md の更新対象セクション
- ...
```

### STEP 2: 反映案を出す

`PROJECT_MEMORY.md` に入れる内容を、セクションごとに差分案として提示する。

この時点では、まだファイルを編集しない。

```markdown
## 📝 PROJECT_MEMORY 反映案

### 更新するセクション
- 1. プロジェクト概要
- 2. Hard Boundaries
- 3. SSoT
- 4. DB設計・計算系データ

### まだ埋めないセクション
- RLS・権限（DB設計確定後）
- テストデータ制約（運用確認後）
```

### STEP 3: ユーザー承認を待つ

以下のような明示承認があるまで、`PROJECT_MEMORY.md` を編集しない。

- `OK`
- `進めて`
- `反映して`
- `この内容で作って`

### STEP 4: `PROJECT_MEMORY.md` に反映する

承認後、以下のルールで更新する。

- 確定済みの内容だけを書く
- 未確定項目は `[未確定]` として残す
- DB設計待ちの項目は `[DB設計確定後に記入]` と明記する
- 推測でテーブル名・カラム名・計算式を作らない
- プロジェクト固有の内容を他案件へ流用しない

### STEP 5: 完了レポートを出す

```markdown
## ✅ PROJECT_MEMORY 更新完了

### 反映したセクション
- ...

### 未確定として残したセクション
- ...

### DB設計確定後に追記すべき項目
- ...

### 注意点
- ...
```

## 書き換えルール

### プレースホルダの扱い

`[プロジェクト名]` などのプレースホルダは、確定情報がある場合だけ置き換える。
確定情報がない場合は消さずに残す。

### 既存内容の扱い

既に記入済みの内容がある場合は、勝手に削除しない。
矛盾がある場合は、以下の形でユーザーに確認する。

```markdown
既存内容Aと新規要件Bが矛盾しています。
どちらを正としますか？
```

### DB設計未確定時の扱い

DB設計が未確定の場合は、以下のように書く。

```markdown
| 領域 | SSoT | 役割 | 参照する画面・処理 |
|---|---|---|---|
| `[対象領域]` | `[DB設計確定後に記入]` | `[要件定義中]` | `[対象画面]` |
```

## 禁止事項

- 要件が曖昧なままDBテーブル名を作る
- SSoT関数名を推測で作る
- 未確定の業務ルールを確定済みとして書く
- 既存 `PROJECT_MEMORY.md` を勝手に上書きする
- `PROJECT_MEMORY.md` の内容を、ユーザー承認なしに大量削除する
- プロジェクト固有の個人名・日付・IDを汎用テンプレートへ混ぜる
