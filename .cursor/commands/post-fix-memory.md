# 修正後 PROJECT_MEMORY 追記コマンド

## ゴール

先方からの指摘、出戻り、既存機能の破壊、仕様誤解、DB設計ミス、表示ズレなどを修正した後に、同じ失敗を繰り返さないための知識を `PROJECT_MEMORY.md` に残す。

このコマンドは、原則として **修正完了後に呼び出す**。

```text
/post-fix-memory
今回の修正内容を PROJECT_MEMORY.md に残して
```

## 使うタイミング

以下のいずれかに当てはまる場合は、このコマンドを使う。

- 先方から不具合・表示ズレ・仕様違いを指摘された
- 修正したが、別の既存機能を壊した
- 要件理解が浅く、出戻りが発生した
- DB設計やデータ同期の前提が漏れていた
- 計算ロジックが複数箇所に分散してズレた
- マスタ変更で過去分が塗り替わった
- 画面上の表示とDB上の正データが一致しなかった
- テスト期待値が実装都合に寄りそうになった
- 次回以降、AIに必ず思い出してほしい注意点ができた

## 実行前に必ず読むもの

1. `PROJECT_MEMORY.md`
2. `.cursor/rules/safety.mdc`
3. `.cursor/rules/database.mdc`
4. `.cursor/rules/invariants.mdc`
5. `.cursor/skills/test-integrity-guardrails/SKILL.md`

## 入力として受け取るもの

ユーザーまたは会話履歴から、以下を集める。

- 先方からの指摘内容
- 実際に起きていた不具合
- 原因
- 修正したファイル
- 修正内容
- 実行したテスト
- 残っているリスク
- 次回以降の再発防止ルール

不足している場合は、推測で埋めずに「要確認」とする。

## 出力ステップ

### STEP 1: 修正後レビューを出す

まず、`PROJECT_MEMORY.md` に追記する前に、以下をチャットへ出す。

```markdown
## 🔍 修正後レビュー

### 1. 先方からの指摘
- ...

### 2. 実際に起きていた問題
- ...

### 3. 根本原因
- ...

### 4. 修正内容
- ...

### 5. 影響を受けた既存機能
- ...

### 6. 実行したテスト・確認
- ...

### 7. まだ残るリスク
- ...
```

### STEP 2: PROJECT_MEMORY 追記案を出す

次に、`PROJECT_MEMORY.md` へ入れる追記案を出す。

追記先の基本は以下。

- `## 5. 📜 過去に AI が壊した事例（再発防止メモ）`
- 必要に応じて、関連するドメインルール・不変条件・テスト方針にも追記案を出す

```markdown
## 📝 PROJECT_MEMORY.md 追記案

### 追記先
- `## 5. 📜 過去に AI が壊した事例（再発防止メモ）`

### 追記内容
- YYYY-MM-DD: [事象]
  - 原因: ...
  - 再発防止: ...
  - 関連ファイル: ...
  - 関連テスト: ...
```

### STEP 3: ユーザー承認を待つ

以下の明示承認があるまで、`PROJECT_MEMORY.md` を編集しない。

- `OK`
- `追記して`
- `反映して`
- `この内容で残して`

### STEP 4: PROJECT_MEMORY.md に追記する

承認後、以下のルールで追記する。

- 既存内容を削除しない
- 既存見出しを勝手に変更しない
- 追記は簡潔にする
- 「誰が悪いか」ではなく「なぜ起きたか」を書く
- 再発防止は、ルール・テスト・DB制約・設計図更新のいずれかに落とす
- 推測を確定事実として書かない

### STEP 5: 完了レポートを出す

```markdown
## ✅ PROJECT_MEMORY.md 追記完了

### 追記した内容
- ...

### 追記しなかった内容
- ...

### 次回以降AIが注意すべきこと
- ...
```

## 追記フォーマット

`PROJECT_MEMORY.md` の過去事例欄には、原則として以下の形で追記する。

```markdown
- YYYY-MM-DD: [短い事象名]
  - 事象: [何が起きたか]
  - 原因: [なぜ起きたか]
  - 再発防止: [次回どう防ぐか]
  - 関連: `[file]`, `[test]`, `[rule]`
```

## 判断基準

### PROJECT_MEMORY.md に追記すべき

- 同じミスを繰り返すと業務上の損害がある
- DB・計算・権限・確定値・手動修正に関わる
- 既存機能を壊した
- ユーザーや先方から明確に指摘された
- テスト追加だけでは防ぎきれない業務知識がある

### PROJECT_MEMORY.md ではなく通常報告でよい

- 単なるtypo
- 一時的な表示崩れで業務ロジックに影響しない
- 既存ルールにすでに十分明記されている
- 再発しても影響が小さい軽微な文言修正

## 禁止事項

- 修正直後に、検証なしで「再発防止済み」と書く
- テストしていないのに「テスト済み」と書く
- 先方指摘を曖昧に丸めて書く
- AI都合で原因を美化する
- `PROJECT_MEMORY.md` をユーザー承認なしに直接編集する
- 大量の会話ログをそのまま貼る
- 個人情報や秘匿情報を不要に残す
