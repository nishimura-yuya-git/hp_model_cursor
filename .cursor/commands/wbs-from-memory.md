# PROJECT_MEMORY → ガントチャートHTML生成コマンド

## ゴール

`PROJECT_MEMORY.md` に蓄積された要件定義、業務ルール、Hard Boundaries、SSoT、未確定事項を読み取り、開発開始前に顧客へ見せられる **ガントチャートHTML** を生成する。

最終成果物は、Googleスプレッドシートへ貼り付けても罫線・日付列・淡い緑のガントバーが残りやすい、コピー機能付きHTMLとする。

Excel / CSV / xlsx ファイルは生成しない。ブラウザで開ける単体HTMLを正とし、HTML内の「ガント表をコピー」ボタンからスプレッドシートへ貼り付ける。

## 最重要ルール

### 1. 生成前の逆質問を必須にする

`/wbs-from-memory` が呼ばれたら、HTMLを生成する前に必ず開始日を確認する。

ユーザー入力に開始日が含まれていない場合は、作業を止めて次のように質問する。

```markdown
ガントチャートの開始日を教えてください。

例: 2026-07-01

開始日をもとに `doc/YYYY.MM.DD/` を作成し、その中にコピー機能付きガントHTMLを保存します。
```

開始日が不明なまま、今日の日付やAIの推測でHTMLを生成してはならない。

### 2. 開始日を受け取ったら保存まで行う

開始日が確定したら、チャット出力だけで終わらせず、HTMLファイルを生成して保存する。

保存先は開始日をもとに以下とする。

```text
doc/YYYY.MM.DD/
```

例:

```text
開始日: 2026-07-01
保存先: doc/2026.07.01/
```

該当ディレクトリが存在しない場合は作成する。

ファイル名は原則として以下にする。

```text
<project_slug>_initial_gantt_wbs.html
```

プロジェクト名が判断できない場合は以下にする。

```text
initial_gantt_wbs.html
```

既に同名ファイルがある場合は、上書き前に確認する。

### 3. 担当は初期値として全て「西村」にする

ガント表の `担当` 列は、ユーザーが別担当者を明示しない限り、全タスクで `西村` と記載する。

役割名だけの `PM`、`開発`、`フロント`、`DB`、`QA` などを担当列に入れない。

必要ならタスク名やメモ側に役割を含めてもよいが、担当列は `西村` に統一する。

### 4. 完成形はガントチャートHTMLにする

通常のWBS表だけで終わらせない。

以下を必須にする。

- 左側に `No`, `工程`, `作業項目`, `担当`, `開始日`, `終了日`
- 右側に開始日からの日付列
- タスク期間に淡い緑のバーセル
- 顧客に見せても違和感がない余白、罫線、見出し
- `ガント表をコピー` ボタン
- コピー成功メッセージ
- Googleスプレッドシートへ貼り付けやすい `text/html` と `text/plain` のクリップボード対応
- ブラウザ表示とスプレッドシート貼り付けの両方を想定したインラインstyle


## 入力として受け取るもの

ユーザーは以下のいずれかを渡す。

```text
/wbs-from-memory
/wbs-from-memory 開始日: 2026-07-01
/wbs-from-memory @PROJECT_MEMORY.md から初期ガントを作って
/wbs-from-memory この追加要件も含めて
```

開始日が含まれていない場合は、必ず逆質問してから生成する。

## 日付の扱い

### 指定がある場合

ユーザーが開始日を指定した場合、その開始日から日付列を作る。

期間は、タスク数と粒度に応じて30日〜60日程度を初期案とする。ただし、これは工程整理用の仮配置であり、実工数の確定ではない。

HTML内の注記に以下を明記する。

```text
日付はユーザー指定の開始日をもとにした表示用の初期案です。実際の開始日・終了日は調整してください。
```

### 指定がない場合

HTMLを生成してはならない。必ず開始日を質問する。

AIが以下をしてはならない。

- 今日の日付から勝手に開始日を決める
- 一般的な工数感から開始日や納期を推測する
- 「来週」などの曖昧な表現を確定日付に変換する
- 人数・稼働日を仮定して逆算する

## WBS生成ステップ

### STEP 0: 開始日確認

開始日が入力に含まれているか確認する。

含まれていない場合は、開始日を質問して停止する。

含まれている場合のみ、以降へ進む。

### STEP 1: PROJECT_MEMORY.md から根拠を抽出する

以下を読み取る。

- プロジェクト概要
- 業務コア
- Hard Boundaries
- SSoT
- DBの事実テーブル / 確定テーブル / 集計テーブル
- RLS・権限境界
- 手動修正・確定済みデータの保護
- 過去に壊れた事例
- 未確定事項
- テスト・運用制約
- 初期リリース必須範囲

### STEP 2: フェーズに分解する

原則として、以下のフェーズ順で整理する。

1. 要件確認・未確定事項整理
2. DB設計・権限設計
3. 業務フロー・SSoT設計
4. 実装準備・変更契約
5. フロントエンド実装
6. API / Supabase / Edge Function 実装
7. データ移行・初期データ整備
8. テスト・不変条件確認
9. ブラウザ動作確認
10. リリース準備・運用確認

該当しないフェーズは無理に入れない。
`PROJECT_MEMORY.md` から判断できないフェーズは、タスク名か注記で要確認と分かるようにする。

### STEP 3: タスク単位に分解する

1タスクは、成果物と完了条件が明確になる粒度にする。

悪い例:

```text
実装する
```

良い例:

```text
事実データ保存後に確定データの再計算が呼ばれることを確認し、手動修正・確定済みデータの保護を維持する
```

### STEP 4: 依存関係とリスクを反映する

以下は必ずWBSに反映する。

- DB設計が決まらないと進められない実装
- RLS・権限設計が必要な機能
- SSoTに関わる計算系
- 手動修正・確定済みデータを保護する必要がある箇所
- 過去に壊れたことがある処理
- 業務フロー設計図との照合が必要な箇所

ガント表は表示幅の都合があるため、長いリスクや根拠は本文下の注記、またはHTML内コメントとして保持してよい。

### STEP 5: ガント用のタスク配列を作る

HTML内では、以下のようなタスク配列を作る。

```js
const tasks = [
  { no: '1.1', phase: '要件確認', task: 'PROJECT_MEMORY整理', owner: '西村', start: 1, end: 3 },
  { no: '1.2', phase: '要件確認', task: '現行HTML棚卸し', owner: '西村', start: 2, end: 5 }
];
```

必須:

- `owner` は全て `西村`
- `start` / `end` は開始日からの相対日数にする
- タスクが重なってもよい
- 無理に人数や稼働日を計算しない
- タスク名はスプレッドシートで読める短さにする

### STEP 6: コピー機能付きHTMLを生成する

HTMLには必ず以下の処理を入れる。

- `renderTable()`
- `buildPlainText()`
- `copyTable()`
- `navigator.clipboard.write()` による `text/html` と `text/plain` のコピー
- clipboard APIが使えない場合の `document.execCommand('copy')` フォールバック

コピー後のメッセージは以下を基本とする。

```text
コピーしました。スプレッドシートのA1へ貼り付けてください。
```

## ガント表の列定義

ガント表は、以下の列を左側に固定的に出す。

| 列 | 内容 |
|---|---|
| No | 連番。例: `1.1`, `2.1` |
| 工程 | 要件確認、DB設計、画面実装など |
| 作業項目 | 実施する作業。短く書く |
| 担当 | 原則 `西村` |
| 開始日 | 開始日から算出した日付 |
| 終了日 | 開始日から算出した日付 |

右側には、開始日から30〜60日分の日付列を出す。

日付列のラベルは `M/D` 形式を基本にする。

例:

```text
7/1, 7/2, 7/3
```

## 保存先ルール

開始日が `2026-07-01` の場合、保存先は以下。

```text
doc/2026.07.01/
```

ディレクトリがなければ作成する。

ファイル名はプロジェクト名からslugを作れる場合、以下。

```text
doc/2026.07.01/compass_bridge_initial_gantt_wbs.html
```

slugが判断できない場合、以下。

```text
doc/2026.07.01/initial_gantt_wbs.html
```

既に同名ファイルがある場合は、上書きせず確認する。

### コピー仕様

`copyTable()` は以下の形式を使う。

```js
await navigator.clipboard.write([
  new ClipboardItem({
    'text/html': new Blob([html], { type: 'text/html' }),

  })
]);
```

フォールバックとして、`document.createRange()` と `document.execCommand('copy')` を用意する。

## HTML出力時の注意

- スプレッドシート貼り付け用の表本体は `rowspan` / `colspan` を使わない。
- 表内セルにはインラインstyleを付け、コピー後も罫線と背景色が残りやすくする。
- 表示用CSSは `<style>` に書いてよい。
- コピー対象の `table.outerHTML` には、セルごとのインラインstyleが入るようにする。
- 重要な業務リスクや根拠を表に詰め込みすぎない。顧客向けの見やすさを優先する。
- 詳細リスクは必要に応じてHTML下部の注記に短くまとめる。

## チャット出力時の形式

保存後、チャットには以下を簡潔に出す。

```markdown
`doc/YYYY.MM.DD/<file>.html` に保存しました。

ブラウザで開いて `ガント表をコピー` を押し、GoogleスプレッドシートのA1へ貼り付けてください。
```

HTML全文をチャットに貼り付ける必要はない。保存したファイルを正とする。

## 禁止事項

- `PROJECT_MEMORY.md` をこのコマンドで編集しない
- 開始日未確認のままHTMLを生成しない
- 今日の日付を勝手に開始日にしない
- 担当列に `PM`、`開発`、`DB`、`QA` などを入れない
- 人数・稼働日を勝手に列追加しない
- DBテーブル名、RLS、SSoT関数を推測で作らない
- 未確定要件を確定タスクとして扱わない
- Excel / CSV / xlsx の生成を最終成果物にしない
- 通常WBS表だけで完了扱いにしない
- コピー機能なしのHTMLで完了扱いにしない
- スプレッドシート貼り付け時に崩れやすい複雑なHTMLを表本体に使わない
- `PROJECT_MEMORY.md` のプロジェクト固有情報を別案件へ無条件コピーしない

```text
□ PROJECT_MEMORY.md を読んだか

□ 開始日をユーザーに確認したか
□ doc/YYYY.MM.DD/ を作成または確認したか
□ 同名HTMLの上書き確認をしたか
□ 担当列が全て「西村」になっているか
□ Hard Boundaries をWBSのリスクに反映したか
□ DB/RLS/SSoTに関わるタスクを分離したか
□ 未確定事項を断定していないか
□ ガント表になっているか
□ 日付列と淡い緑のバーがあるか
□ コピー機能があるか
□ text/html と text/plain のコピーに対応しているか
□ 保存後、ファイルを読み戻したか
```

## 呼び出し例

### 開始日なし

```text
/wbs-from-memory
```

この場合は生成せず、開始日を質問する。

### 開始日あり

```text
/wbs-from-memory 開始日: 2026-07-01
```

この場合は `doc/2026.07.01/` を作成し、コピー機能付きガントHTMLを保存する。

### 追加要件あり

```text
/wbs-from-memory 開始日: 2026-07-01
この追加要件も含めてWBS化して
```

追加要件もタスク分解に含め、同じガント形式で保存する。
