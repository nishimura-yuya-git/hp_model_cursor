# LINE連携開発マニュアル（汎用）

## 目的

LINE Messaging API、LINEログイン、Webhook、自動通知などを実装するときに、誤送信・無限ループ・認証情報漏洩を防ぎながら安全に開発する。

---

## 1. 着手前に確認すること

実装前に、以下をユーザーへ確認する。

```text
□ LINE連携の目的（通知 / ログイン / 問い合わせ / 予約 / リマインド等）
□ 対象ユーザー範囲（全員 / 特定ロール / テストユーザーのみ）
□ テスト送信してよいLINEユーザーID
□ 一斉送信・ブロードキャストを許可する条件
□ Webhook URLの公開先
□ 本番環境と検証環境の切り分け
□ 送信ログ・失敗ログの保存方針
```

環境変数はプロジェクトごとに確認する。

```text
LINE_CHANNEL_ID
LINE_CHANNEL_SECRET
LINE_CHANNEL_ACCESS_TOKEN
LINE_WEBHOOK_SECRET（必要な場合）
LINE_LOGIN_CHANNEL_ID（LINEログインを使う場合）
LINE_LOGIN_CHANNEL_SECRET（LINEログインを使う場合）
```

## 2. 安全なテスト方針

開発初期は、必ずテスト対象ユーザーを限定する。

```ts
const ALLOWED_TEST_LINE_USER_IDS = new Set([
  process.env.LINE_TEST_USER_ID,
].filter(Boolean));
```

### 必須ルール

- テスト完了まで、全ユーザーへの一斉送信は実装しても実行しない。
- ブロードキャスト処理には、環境変数や管理画面による明示的な実行ガードを設ける。
- テスト送信先は、ユーザーが許可したLINEユーザーIDだけに限定する。
- 本番データを使う場合は、送信対象と文面を実行前に必ず確認する。

---

## 3. Webhook実装の注意

Webhookでは以下を必ず守る。

- 署名検証を行い、LINEからの正当なリクエストだけ処理する。
- ボット自身のイベントや重複イベントに反応して無限ループしないようにする。
- 同じイベントIDを二重処理しないため、必要に応じて処理済みイベントを記録する。
- 返信・Push・Multicastの用途を分け、意図しない大量送信を防ぐ。
- 失敗時は指数バックオフや再試行回数上限を設ける。

---

## 4. DB設計の考え方

LINE連携でDBを使う場合、以下の分離を検討する。

| 種別 | 例 | 注意点 |
|---|---|---|
| ユーザー紐付け | `line_user_id`, `profile_id`, `linked_at` | 本人確認・解除手段を用意する |
| 送信ログ | 送信先、テンプレート、結果、エラー | 個人情報とログ保存期間に注意 |
| Webhookログ | event_id, type, received_at | 二重処理防止に使える |
| 通知設定 | 通知種別ごとのON/OFF | ユーザーの同意を尊重する |

RLSやAPI認可が必要な場合は、実装前に権限境界を設計する。

## 5. 実装後の確認

```text
□ テストユーザーだけに送信されるか
□ 署名検証が失敗したリクエストを拒否するか
□ 同じWebhookイベントを二重処理しないか
□ 一斉送信に明示的なガードがあるか
□ エラー時に無限リトライしないか
□ 認証情報がコード・ログ・コミットに出ていないか
□ 送信ログから原因調査できるか
```
