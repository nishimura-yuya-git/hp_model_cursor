# RAG資料取り込み（PDF/PPTX）における技術的教訓とベストプラクティス

RAG（Retrieval-Augmented Generation）システムにおいて、PDFやPPTXなどの資料を取り込む際の不具合分析と、安定稼働のための実装指針です。

## 1. トラブル分析：なぜ「文字起こし」が失敗し、どう解決したか

### 1.1 ストレージ・パスの「Invalid key」エラー
*   **原因**: 日本語や特殊文字を含むファイル名をそのままクラウドストレージ（Supabase Storage等）のキーとして使用したため、パス形式の制限に抵触した。
*   **解決**: ストレージパスをファイル名に依存させず、UUIDを用いた固定形式（例: `docId/source.pdf`）に変更した。

### 1.2 画像ベースPDFによる抽出漏れ
*   **原因**: `pdfjs-dist` などの標準的なテキスト抽出ライブラリは、文字が「画像」として埋め込まれたPDF（スキャン資料等）からテキストを取得できない。
*   **解決**: テキスト抽出量が極端に少ない場合に、Gemini Vision API（OCR）へフォールバックする仕組みを導入した。

### 1.3 モデルの可用性（404エラー）
*   **原因**: `gemini-2.0-flash` などのプレビュー版モデルを使用したが、ユーザーのAPIキーや地域設定によっては利用不可（404）となる場合があった。
*   **解決**: 安定版かつ汎用性の高い `gemini-1.5-flash` をデフォルトとして採用した。

### 1.4 UXの「処理中」停滞
*   **原因**: テキスト抽出後の「ベクトル化（Embedding）」処理を逐次（await）実行していたため、文字数が多い場合にフロントエンドがタイムアウトまたはフリーズしたように見えた。
*   **解決**: ベクトル化を非同期（バックグラウンド）処理に切り出し、チャンク保存が完了した時点で即座にステータスを `ready` に更新するようにした。

## 2. 実装のベストプラクティス

### ストレージ管理
- **パスの正規化**: ユーザー入力を避け、UUID等の安全な識別子をパスに用いる。
- **自動クリーンアップ**: DB保存に失敗した場合は、Storageにアップロード済みのファイルも確実に削除するロジックを組む。

### テキスト抽出（OCR）
- **DBキャッシュ**: 抽出したテキスト全体をテーブル（`extracted_text`等）に保存する。これによりプレビュー時のGemini再実行を避け、コストと待ち時間を削減する。
- **手動再生成**: OCRの精度が低い場合に備え、ユーザーが明示的に再抽出を要求できるボタンを用意する。

### 進捗の視覚化
- **フェーズ別の詳細表示**: 「アップロード」「AI抽出」「DB保存」などの各ステップをダイアグラムで示し、ユーザーに進行状況を伝える。
- **リアルタイム更新**: 0%から100%までの数値をリアルタイムに反映させ、システムが動作していることを保証する。

### RAG検索の最適化
- **非同期ベクトル化**: ユーザーの「取り込み完了」体験を優先し、Embedding処理はバックグラウンドで行う。
- **カテゴリ制限**: AIが誤ったコンテキストを参照しないよう、チャットで参照可能なカテゴリを環境変数等で制御（Allowlist方式）する。
