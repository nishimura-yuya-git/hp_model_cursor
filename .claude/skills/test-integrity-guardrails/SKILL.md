# テスト整合性ガードレール

## 目的

バグ修正時に、以下を禁止する。

- 自分が書いたコードに合わせてテスト期待値を甘くする
- 実画面や実データと矛盾しているのに、テストだけ通る状態で完了扱いにする
- 原因未特定のまま、通りやすい単体テストだけを足して安心する

## 最優先の原則

1. **正解はコードではなく外部証拠で決める**
2. **テストは実装を守るためではなく、事実を固定するために書く**
3. **ユーザーのスクリーンショット、実DB値、既存の別画面の正しい表示は強い証拠として扱う**
4. **期待値を変更する前に、「なぜその値が正しいか」を説明できなければ変更しない**

## 適用トリガー

以下のときはこのスキルを適用する。

- バグ修正
- 回帰テスト追加
- 「画面ではこう見えるのにテストは通る」系の不整合
- 表示金額、時給、交通費、集計値などの計算差異調査
- 既存テストの期待値を書き換えたくなったとき

## 修正前チェック

実装前に、次を必ずそろえる。

- **ユーザー報告の事実**: 日付、対象者、画面、期待値、実際値
- **外部証拠**: スクリーンショット、DBレコード、別画面の表示、CSV/Excel、仕様メモ
- **計算式の根拠**: どの値を何に掛けるか、どこで切り捨てるか、交通費や手当の扱い
- **実際の表示経路**: どの画面がどの関数を通って値を出しているか

## テスト作成ルール

### 必須

- テストケース名に、実際のバグの性質を入れる
- 期待値は、実装ではなく外部証拠から決める
- 1本は**今回の不具合をそのまま再現するケース**を入れる
- 可能なら、同じ値が出る**画面経路違い**も確認する

### 禁止

- 「今のコードの出力」に期待値を合わせる
- 原因が分からないまま期待値だけ更新する
- 丸め、手当、手動入力、null/0 の扱いを省いた単純化ケースだけで済ませる
- UI不具合なのに、UIで使われていない関数だけテストして完了にする

## 期待値変更のルール

既存テストの期待値を書き換えてよいのは、次を全部満たす場合だけ。

- 外部証拠で旧期待値が誤りだと示せる
- 変更理由を日本語で説明できる
- 期待値変更後も、ユーザー報告ケースを新規または既存テストで捕まえられる
- 実画面または実データ経路で確認した

1つでも満たせないなら、期待値ではなく実装か理解が間違っている可能性を疑う。

## 実装修正ループ

1. ユーザー報告を箇条書きで固定する
2. 実データと画面表示を突き合わせる
3. 実際の表示経路を特定する
4. その経路を再現する回帰テストを追加する
5. テストを失敗させてから実装する
6. 修正後、関連テストを実行する
7. 可能ならブラウザで対象画面を確認する
8. **テストは通るが画面が違う** 場合は完了扱いにしない

## 画面差異バグの追加ルール

表示不具合では、次を順に疑う。

- 本体計算ロジックと一覧表示ロジックが二重実装されていないか
- `null` と `0` を別扱いすべき箇所が混ざっていないか
- 保存済みサマリ値を再利用していないか
- 時給表示と支給合計が別経路になっていないか
- `shift_id` なし打刻やフォールバック経路が別扱いになっていないか

## レビュー前セルフチェック

以下をすべて確認する。

- [ ] ユーザー報告の値をそのまま再現するテストがある
- [ ] 期待値の根拠を説明できる
- [ ] 既存テストの期待値を変えた場合、その理由を説明できる
- [ ] 実装に都合よくケースを単純化していない
- [ ] UI表示と同じ経路を少なくとも1つ検証した
- [ ] 「テストは通るが画面は違う」を放置していない

## レポート時の必須記載

バグ修正後は、次を短く報告する。

- **何が正解値か**
- **どの外部証拠で正解を判断したか**
- **どの表示経路が未修正だったか**
- **追加した回帰テストが何を固定しているか**
- **実画面確認をしたか、していないなら何がブロッカーか**

## このスキルの意図

「テストが通った」ことより、
**「ユーザーが見ている不具合が本当に消えた」こと** を優先する。
