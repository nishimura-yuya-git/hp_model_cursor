# 影響範囲・デグレード検証コマンド

## ゴール
- 最近の修正内容を分析し、依存関係や共通処理への影響を洗い出す。
- 修正した箇所以外で「動かなくなる可能性がある場所」を特定し、検証（またはテスト実行）を行う。

## 検証ステップ（実行時にAIが辿るべき手順）

### 1. 変更内容の特定
- 直近のコミット、または `git diff` を使用して変更されたファイルを特定する。
- どの関数、コンポーネント、または変数が変更されたかを抽出する。

### 2. 依存関係の分析（静的解析）
- **インポートの追跡**: 変更されたファイルやシンボル（関数等）が他のどのファイルからインポートされているかを `Grep` で検索する。
- **共有処理の確認**: `AuthContext`, `utils`, `services` などの共通モジュールを変更した場合、それを利用している全画面をリストアップする。
- **型の整合性**: `npx tsc --noEmit` 等を実行し、型定義の変更が他のファイルでエラーを引き起こしていないか確認する。

### 3. 動的検証・E2Eテスト
- **Playwright MCP の活用**: 影響を受ける可能性がある主要な画面（ログイン、一覧表示、申請フロー等）を Playwright で実際に操作し、期待通りに動作するか確認する。
- **コンソールエラーの確認**: ブラウザ実行中にエラーや警告が出ていないかチェックする。

### 4. チェックリストによる最終確認
以下の観点で副作用がないか自己点検する：
- [ ] **認証・認可**: ログイン状態や権限チェックが全画面で正しく動作するか。
- [ ] **キャッシュ**: `localStorage` や状態管理（Context）の変更が、再ログイン後やリロード後も正常か。
- [ ] **共通UI**: ヘッダー、ナビゲーション、トースト通知などの共通部品が崩れていないか。
- [ ] **パフォーマンス**: 無限ループや不要な再レンダリングが発生していないか。

## 出力フォーマット
検証結果を以下の形式で報告してください：

### 🔍 影響範囲の分析結果
- **修正箇所**: [ファイルパス/シンボル名]
- **直接的な影響**: [修正した機能の動作確認結果]
- **間接的な影響範囲**: [この修正により挙動が変わる可能性がある他の画面や機能]

### 🧪 検証ログ（実施したテスト）
- [ ] [検証内容1]: [結果]
- [ ] [検証内容2]: [結果]

### ✅ 結論
- [副作用なし / 以下の懸念あり]
- **理由**: [根拠となる分析結果やテストエビデンス]
- **推奨アクション**: [追加の修正や注視すべき点]
