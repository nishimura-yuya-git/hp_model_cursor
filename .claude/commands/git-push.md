# Git Diff → Commit → Push コマンド

## 🚨 絶対禁止事項

- **`--force` / `--force-with-lease` によるforce pushは絶対禁止**
- **`--no-verify` によるフック無効化は絶対禁止**
- **`.env` や認証情報ファイルをコミットしない**
- **`git config` の変更は絶対禁止**

---

## 実行手順

### STEP 1: 現在の状態を確認

```bash
git status
```

- 変更・追加・削除されたファイルを把握する
- Untracked files（新規ファイル）も確認する

---

### STEP 2: 差分の詳細を確認

```bash
# 変更ファイルの統計（追加・削除行数）
git diff --stat

# 最近のコミット履歴（コミットメッセージのスタイルを参考にする）
git log --oneline -5
```

主要ファイルの差分内容も確認し、**何が変わったのか**を正確に把握する。

```bash
# 特定ファイルの差分確認（必要に応じて）
git diff <ファイルパス>
```

---

### STEP 3: 変更内容を分析してコミットメッセージを作成

差分から以下を整理する：

| 分類 | 確認ポイント |
|------|------------|
| **新機能** | 新規ファイル・新規コンポーネント・新規ページ |
| **改善** | 既存機能の修正・リファクタリング・パフォーマンス改善 |
| **バグ修正** | エラー修正・不具合対応 |
| **設定変更** | package.json・設定ファイルの変更 |
| **削除** | 不要ファイル・廃止機能の削除 |

**コミットメッセージのフォーマット：**

```
<タイトル（50文字以内・日本語）>

## 主な変更内容

### <カテゴリ名>
- <変更点1>
- <変更点2>

### <カテゴリ名>
- <変更点>
```

**タイトルの接頭辞ルール：**
- 新機能追加 → `〇〇機能の実装`
- 既存機能改善 → `〇〇の改善・修正`
- バグ修正 → `〇〇のバグ修正`
- リファクタリング → `〇〇のリファクタリング`
- 設定・環境 → `〇〇の設定追加・更新`

---

### STEP 4: 全変更をステージングしてコミット

```bash
# 全変更をステージング（新規ファイル含む）
git add -A

# ステージング状態を再確認
git status

# コミット（HEREDOCでメッセージを渡す）
git commit -m "$(cat <<'EOF'
<タイトル>

## 主な変更内容

### <カテゴリ>
- <変更点>

EOF
)"
```

---

### STEP 5: mainブランチにpush

```bash
git push origin main
```

---

### STEP 6: push完了の確認

push成功後、以下を報告する：

```
## コミット内容サマリー

**コミットID**: `<hash>`
**変更ファイル数**: XX ファイル（XX 行追加 / XX 行削除）

### 主な変更内容

| カテゴリ | 内容 |
|----------|------|
| 新機能 | ... |
| 改善 | ... |
| 削除 | ... |
```

---

## ⚠️ 注意事項

### コミットしてはいけないファイル
- `.env` / `.env.local` / `.env.production` などの環境変数ファイル
- `node_modules/` ディレクトリ
- 認証情報・APIキーが含まれるファイル
- ビルド成果物（`dist/`）

### コミットメッセージのポイント
- **「何を」ではなく「なぜ・何のために」を意識する**
- 日本語で記述する（プロジェクトルール準拠）
- タイトルは50文字以内に収める
- 箇条書きで変更点を明確に列挙する

### ブランチ確認
- 必ず `main` ブランチにいることを確認してからpushする
- `git status` の1行目で `On branch main` を確認する
