# UI Polish Loop

## Goal

ユーザーが添付した見本画像、理想デザイン、指示文の意図を抽出し、一発目のUI出力品質を上げる。実装後は「完成と言ってよい根拠」を確認し、不足があれば小さく修正を回す。

## Input

以下をそのまま渡してよい。

```text
UI Polish Loopで進めて。

対象画面:
（分かればURLやファイル）

理想画像:
（添付画像）

指示:
（ユーザーやお客さんの文章）
```

## Required First Step

コードを書く前に、必ず以下を言語化する。

### 1. 画像から抽出すること

- レイアウト構造: 何が上/中央/下、左右、主従関係か。
- 余白: 広い余白か、高密度か、セクション間の呼吸感。
- 視線誘導: 最初に見せたい要素、次に押させたい操作。
- 色: 主色、背景色、状態色、強調色。
- 角丸・影・境界線: 丸み、立体感、静かな高級感。
- タイポグラフィ: 見出し、本文、補助文字のサイズ差。
- ボタン: 押したくなるか、主副の階層が明確か。
- 画像/キャラクター: 顔、持ち物、伝えたい要素が見切れていないか。
- モバイル/PC差: 片方の縮小で済ませていないか。
- 禁止事項: Lucide等の禁止アイコン、装飾英語、過剰アニメーション、チープな影。

### 2. 意図と文脈

- ユーザーは何を「良い」と感じているか。
- 見本画像のどこを真似るべきで、どこはプロジェクト文脈に合わせて変えるべきか。
- 業務UIか、HP/LPか。
- 操作導線を増やすべきか、説明として見せるべきか。

### 3. 完成判定

実装前に、完成条件を3〜7個に絞る。

例:

- 主要操作が1秒で分かる。
- 見本画像と同じ余白・重心・視線誘導になっている。
- モバイルで見切れない。
- 画像の顔や重要アイテムが見える。
- 日本語文言だけで意味が伝わる。
- 禁止アイコンライブラリを使っていない。

## Implementation

- 既存コンポーネントを優先する。
- 一度で完成扱いにしない。
- 実装後にブラウザ表示、スクリーンショット、または差分確認を行う。
- 見本画像との差分を再度言語化する。
- 不足があれば余白、サイズ、位置、文言、画像位置を小さく修正する。

## Stop

以下の場合は停止して確認する。

- 添付画像の意図が読み取れない。
- 業務UIとHP/LPのどちらのルールを適用すべきか不明。
- 見本の再現がプロジェクト禁止事項に反する。
- 必要な画像素材が存在しない。
- 既存UI導線を大きく変える必要がある。

## Evaluation

標準評価は以下。

```bash
pnpm run loop:run
pnpm run type-check
pnpm run check:hard-boundaries
pnpm run check:architecture
```

画面変更では、必要に応じてブラウザ確認・スクリーンショット確認を行う。

## Output

- 画像から抽出した意図
- 現状との差分
- 完成判定
- 実装内容
- 確認した表示ポイント
- 未確認項目と理由
